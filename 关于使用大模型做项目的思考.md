# 使用大模型做项目的完整指南

> 工具链：Claude.ai + Claude Code + GitHub Desktop + GitLab
> 核心定位：学习的不是如何编程，而是如何驾驭 AI 工具实现想要的功能

---

## 第一部分：用大模型做项目的思路框架

### 两种模式对比

在实际工作中存在两种主流的 AI 开发模式：

**模式 A：大批量生成模式（效率优先）**

把需求文档和现有代码一起喂给 Claude Code → 让它分析未实现的功能 → 一次性补齐 → 只要页面不报错就继续 → 再开新窗口重复扫描补齐，直到功能完整。

- 优点：极大提高开发效率，淡化开发工作量
- 缺点：依赖 AI 编程能力，可能出现"不报错但业务逻辑不对"的问题
- 适用场景：你已经理解业务逻辑的功能、简单的 CRUD、重复性高的页面

**模式 B：拆步推进模式（可控优先）**

先设计方案 → 拆成小模块 → 逐个实现 → 每步验证 → 迭代优化。

- 优点：过程可控，出了问题容易定位
- 缺点：速度相对慢
- 适用场景：不熟悉的业务逻辑、复杂的多模块联动、涉及权限/审批流等核心功能

**实际工作中的建议：混合使用，不要二选一。** 简单的用模式 A 冲效率，复杂/核心的用模式 B 保质量。随着经验增长，模式 A 的适用范围会越来越大。

---

### 不管用哪种模式，都要遵守的流程

#### 第一步：自己先想清楚（10-15 分钟）

拿到任务后不要急着开对话。先用自己的话回答三个问题：

- **输入是什么？** 这个功能接收什么数据/操作？
- **输出是什么？** 最终要实现的效果是什么样的？
- **大致思路是什么？** 哪怕很粗糙，写下来。

你给的上下文越清晰，AI 给你的结果就越准确。这是最重要的一条原则。

#### 第二步：用 Claude.ai 做高层设计

在写代码之前，先在 Claude.ai 网页端把技术方案聊清楚，拿到一个清晰的路线图。这一步不写代码，只讨论方案。

#### 第三步：用 Claude Code 做具体实现

方案确定后切到 Claude Code 落地。核心要点：

- 用 `@文件名` 引用相关代码文件，提供真实上下文
- 模式 A：直接让 Claude Code 分析需求文档 + 现有代码，一次性补齐未实现功能
- 模式 B：拆成小模块逐步实现，每完成一个先验证再继续

#### 第四步：提交前自测（关键步骤，10 分钟）

无论用哪种模式，提交前都要自己按需求文档**手动走一遍核心流程**。不是看代码，而是在页面上操作。这个成本很低，但能拦住大部分"不报错但业务不对"的问题。

重点关注：
- 页面是否有 404 或明显报错
- 核心业务流程是否和需求文档一致（审批顺序、数据流转、权限控制等）
- 边界情况（空数据、异常输入）

#### 第五步：出了问题让 AI 帮你定位

报错时不要自己硬啃，直接把完整错误信息喂给 Claude Code，让它扫描项目文件定位问题。关键是提供清晰的错误描述："执行了什么 → 期望什么 → 实际什么"。

#### 第六步：总结归档（5 分钟）

每完成一个任务记录：哪些 prompt 效果好、踩了什么坑、学到了什么。

## 第二部分：写出有效 Prompt 的核心习惯

### 核心公式：背景 + 目标 + 约束 + 期望格式

| 要素 | 说明 | 示例 |
|------|------|------|
| **背景** | 项目信息、技术栈、当前进度 | "我在做一个 React+Node.js 的任务管理系统，用户登录已完成" |
| **目标** | 明确要 AI 做什么 | "帮我实现任务的 CRUD 接口，使用 Express + MongoDB" |
| **约束** | 限制条件和要求 | "需要兼容现有的数据库 schema，要包含错误处理" |
| **期望格式** | 希望 AI 怎么回复 | "先给整体思路，再分文件给代码，加中文注释" |

---

### 常见场景 Prompt 模板

**场景一：开始一个新功能**

```
我在做 [项目名称]，技术栈是 [技术栈]。现在需要实现 [功能描述]。
当前已经完成了 [已完成的部分]。
要求：[具体约束条件]。
请先给出整体实现方案，然后分步骤实现。
```

**场景二：大批量补齐功能（模式 A）**

```
请分析 @需求文档 和当前项目代码，对比找出还有哪些功能未实现，
然后把未实现的功能补齐。
要求：保持和现有代码风格一致，不要破坏已有功能。
```

**场景三：调试报错**

```
我执行了 [具体操作]，期望结果是 [期望结果]，但实际报错：
[粘贴完整错误信息]
相关代码在 @[文件路径]。请帮我分析原因并给出修复方案。
```

**场景四：Review 代码**

```
请帮我 review 一下 @[文件路径] 这个文件。重点关注：
1. 有没有潜在的 bug
2. 性能是否有优化空间
3. 代码结构是否合理
```

**场景五：架构设计讨论**

```
我需要设计 [系统/模块名称] 的架构。
需求：[核心需求]。技术约束：[限制条件]。
我的初步思路是 [你的想法]，请帮我评估这个方案，指出问题，并给出你的建议。
```

### 实用技巧速查

| 技巧 | 说明 |
|------|------|
| **给上下文** | 在 Claude Code 中用 `@文件名` 引用相关代码文件，让 AI 看到项目实际情况 |
| **拆小任务** | 不要一次让 AI 写整个功能，拆成小模块逐步实现（模式 B 时） |
| **贴错误信息** | 报错时贴完整错误信息，说明"执行了什么、期望什么、实际什么" |
| **追问为什么** | 对不理解的代码追问"为什么选择这个方案"，提升自己对 AI 的控制力 |
| **中文提问 OK** | Claude 的中文理解能力很强，关键是表达清晰而不是语言本身 |
| **对话太长就新开** | 如果对话超过 20 轮效果变差，新开一个对话并总结之前的进度 |
| **多轮扫描** | 用模式 A 时，一轮可能遗漏功能，多开几个新窗口重复"分析+补齐"直到完整 |

---

### 核心心态

**把 AI 当作一个编程能力极强但不了解你项目的同事。**

你不会对同事说"帮我写后端"，你会说"我需要一个用户认证接口，用 JWT，这是现有的 User model..."。你给的上下文越清晰，AI 给你的结果就越准确。

**你要学的不是写代码，而是驾驭 AI 的能力。** 这种能力的核心是：精准地描述你要什么、出了问题高效地引导 AI 修复、知道什么时候该信任 AI 什么时候该自己验证。
